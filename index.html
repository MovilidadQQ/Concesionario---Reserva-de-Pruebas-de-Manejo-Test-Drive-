Arquitectura de esta página — qué resuelve cada capa

Al ser un único archivo HTML/CSS/JS sin servidor real, el backend y la base de datos se simulan en JavaScript dentro del mismo documento: funciones puras hacen de "endpoints" y un objeto DB persistido en localStorage hace de tablas.

FrontendFormulario + render de tabla

Captura vehículo, asesor, cliente, fecha y franja horaria. Renderiza la grilla de reservas del día leyendo directamente de DB.reservas.

Backend simuladoValidación de cruce de horarios

Antes de crear la reserva, verificarDisponibilidad() recorre la tabla de reservas y chequea solapamiento de intervalos tanto para el vehiculoId como para el asesorId en la misma fecha. Si hay conflicto en cualquiera de los dos, rechaza la operación e informa cuál recurso está ocupado.

function seSuperponen(iniA, finA, iniB, finB) {
  return iniA < finB && iniB < finA;
}

function verificarDisponibilidad(vehiculoId, asesorId, fecha, ini, fin) {
  const delDia = DB.reservas.filter(r => r.fecha === fecha);

  const conflictoVehiculo = delDia.some(r =>
    r.vehiculoId === vehiculoId && seSuperponen(ini, fin, r.ini, r.fin));

  const conflictoAsesor = delDia.some(r =>
    r.asesorId === asesorId && seSuperponen(ini, fin, r.ini, r.fin));

  return { disponible: !conflictoVehiculo && !conflictoAsesor,
           conflictoVehiculo, conflictoAsesor };
}

Base de datos simuladaTablas en localStorage

Tres tablas: vehiculos, asesores y reservas (con vehiculoId, asesorId, cliente, fecha, ini, fin en minutos). Cada reserva exitosa hace push a DB.reservas y persiste con localStorage.setItem, tal como haría un INSERT en SQL.
